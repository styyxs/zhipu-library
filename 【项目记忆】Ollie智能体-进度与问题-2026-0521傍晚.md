### 🔴 当前问题（森哥描述的完整现象）

**场景：** Apple 单词，Step1 → Step2（跟读）→ 用户说错 → 评分失败 → 回到 Step2

**完整流程：**
1. 第一次进 Step2 "太棒了，跟我念一遍好吗？"：✅ 有 Apple 文字 + ✅ 有 TTS 读音
2. 用户按麦克风说话 → 识别 → 评分（score < 0.5）→ 显示"再试一次吧，你可以的！"
3. 短暂停留后，自动回到 Step2 "太棒了，跟我念一遍好吗？"
4. **此时界面：❌ 没有 Apple 文字 + ❌ 没有 TTS 读音（静止界面）**
5. 用户点麦克风按钮 → 按钮变红 → 下一秒报错 `recognition has already started`
6. 麦克风按下又松开 → 短暂出现"再试一次" → 立刻又跳回 Step2 静止界面
7. 再点麦克风 → 再次报 `recognition has already started`（循环）

---

### 🧠 根因分析（2026-05-21 傍晚）

**核心问题：Step3 评分失败后 `goToStep(2)` 重新进入 Step2 时，`speech.js` 的状态没有正确重置**

具体分析：
- Step3 结束时调用 `goToStep(2)` 重新进入 `runStep2()`
- 但此时 `speech.js` 的 `_isStarting`、`_isStopping`、`isListening` 等状态可能处于残留状态
- 当用户点麦克风时，`startListening()` 被调用，但 recognition 实际上处于不可用状态
- `start()` 抛出 `recognition has already started`

---

### ✅ 已实施的4项修复（2026-05-21傍晚，全部在 `js/speech.js`）

**修复1：catch 块里重置 `_isStarting`**
```js
// 如果 start() 抛错，_isStarting 永远卡在 true，导致后续所有 startListening() 被拦截
// 修复：在 catch 里加 this._isStarting = false
```

**修复2：onresult 检查 `_isStopping` 标志**
```js
// stop() 调用后，onresult 可能还会收到 late result
// 导致 goToStep(3) 被调用多次（评分逻辑走两次）
// 修复：在 onresult 开头检查 this._isStopping，true 则忽略
```

**修复3：onend 和 onerror 重置所有标志**
```js
// 识别意外结束时（网络中断、用户切换标签页等），状态没有清干净
// 修复：onend 和 onerror 里同时重置 _isStopping 和 _isStarting
```

**修复4：stopListening() 先设 `_isStopping` 再调 stop()**
```js
// 手动停止时，先标记停止中，再调用 stop()
// 避免 stop() 和 start() 竞争导致的状态不一致
```

### 问题2：网页动画和语音乱跳不稳定

**现象：**
- 森哥打开 `ollie.zhipu-library.top` 后，没有做任何交互
- 页面随机出现：时而冒一句话、时而冒一个动画
- 完全无规律，不可预期

**可能原因分析：**
1. **LocalStorage 进度状态问题：** 之前学习的进度被保存了，刷新页面后自动恢复到了某个学习 Step，自动触发了 TTS 和动画
2. **Worker 请求失败回调问题：** TTS 发出请求后失败，触发了某个 fallback 逻辑，导致异常行为
3. **降级方案行为异常：** Web Speech API 在某些情况下可能自动播放

### 问题3：语音和动画不同步

**现象：**
- 有时先出动画有时先出语音
- callback 机制可能有问题

**代码分析：**
- `app.js` 第 65 行：`TTS.speak(..., { callback: () => this.showStep1Word(word) })`
- 动画是在 callback 里才显示的，不是同步的
- TTS 走降级方案时 callback 时机可能不稳定
